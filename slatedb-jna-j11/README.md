# slatedb-jna-j11 — SlateDB on Java 11/17 via the JNA binding

**Claim proven by this module:** SlateDB's "requires Java 22" floor is a **packaging choice**, not a
limitation of SlateDB. The same native library runs on **JDK 11, 17, and 25** through a different
generated front-end.

## Why the floor exists (and why it's not fundamental)

The official Maven artifact `io.slatedb:slatedb-uniffi` (used by the sibling `slatedb-verify` module)
is generated by [`uniffi-bindgen-java`](https://github.com/IronCoreLabs/uniffi-bindgen-java). That
generator emits **`java.lang.foreign` / FFM (Panama)** code — `Linker`, `SymbolLookup`, `Arena`,
`upcallStub`, one `MethodHandle` per Rust fn — which is finalized only in **JDK 22 (JEP 454)**. The
classes are bytecode major version **66**; a JDK 11/17 `javac` can't even read them.

But UniFFI also has a **Mozilla-standard Kotlin backend that uses JNA**, and **JNA runs on Java 8+**.
Both front-ends call the *same* native `libslatedb_uniffi` C ABI (441 exported symbols). So the JVM
floor moves from 22 down to 8/11 purely by regenerating.

## What this module contains

| Path | What it is |
|---|---|
| `src/main/kotlin/uniffi/slatedb/slatedb.kt` | The regenerated + patched **JNA** binding (package `uniffi.slatedb.*`, distinct from the FFM binding's `io.slatedb.uniffi.*`). 18k lines, machine-generated. |
| `src/main/kotlin/io/slatedb/flink/poc/SlateBridge.kt` | Thin Kotlin adapter turning SlateDB `suspend` fns into **blocking** calls callable from Java (the mandatory "adapter cost" of the JNA route). |
| `src/main/java/io/slatedb/flink/poc/SlateDbJnaJ11Main.java` | Plain **Java** driver: open, 500× put, MEMTABLE flush, detached **checkpoint**, 500× read-back. |

## How the checked-in binding was produced (reproducible)

1. **`git worktree add --detach <wt> v0.14.1`** — generate from the exact release tag so the binding
   matches the *published* native lib. (Clone `HEAD` had drifted past 0.14.1: it added
   `admin_run_gc_once`, a symbol **absent** from the 0.14.1 `.dylib` → `UnsatisfiedLinkError` at
   runtime. Version match is mandatory.)
2. **`uniffi-bindgen` 0.31.1** (matching the workspace `Cargo.lock`):
   `generate --language kotlin --library libslatedb_uniffi.dylib`.
3. **Patch a uniffi-0.31 codegen bug** (7 sites): SlateDB's Rust `Error` variants carry a `message`
   field, so the generator emits a constructor `val \`message\`` that **collides with
   `Throwable.message`** — this does **not compile on any kotlinc** (tried 1.9.24 and 2.4). Fix: make
   it `override val \`message\`` and delete the now-redundant `override val message get() = …`.
   *This is the concrete "fork maintenance tax" of the JNA route — you own this patch forever.*

## Run it

Needs the published native lib on `jna.library.path` (no Rust toolchain to run). Stage it once:

```bash
mkdir -p native
unzip -oj ~/.m2/repository/io/slatedb/slatedb-uniffi/0.14.1/slatedb-uniffi-0.14.1.jar \
  'darwin-aarch64/libslatedb_uniffi.dylib' -d native   # or linux-x86-64/…so, etc.
```

Then, pointing `JAVA_HOME` at JDK **11**, **17**, or **25**:

```bash
mvn -q clean compile
mvn -q exec:java -Djna.library.path="$PWD/native"
```

No `--enable-native-access`, no `--add-opens` — those are FFM requirements; JNA needs neither.

The async completions genuinely exercise the JNA **upcall** path
(`ffi_slatedb_uniffi_rust_future_poll_*` + `UniffiRustFutureContinuationCallback : com.sun.jna.Callback`) —
the exact mechanism the FFM binding implements with `upcallStub`. Including the checkpoint, that's the
async surface Flink needs for exactly-once.

## Trade-offs vs the official FFM binding (`slatedb-verify`)

- **You maintain a fork.** SlateDB publishes only the FFM artifact. Every upgrade = regenerate →
  re-apply the `message` patch → recompile → re-test.
- **Kotlin bridge is mandatory.** `suspend` fns surface as `(args, Continuation)` — not ergonomically
  callable from Java. The FFM binding gives you a clean `CompletableFuture` API for free.
- **JNA marshalling overhead** per call vs FFM `MethodHandle` downcalls — likely negligible next to
  object-storage latency, but real and unmeasured for this workload.

**Recommendation:** if you can run Flink on JDK 22+, use the official FFM binding as-is (zero fork
maintenance). Take this JNA route only if hard-pinned to JDK 11/17. See root `README.md` §17.
